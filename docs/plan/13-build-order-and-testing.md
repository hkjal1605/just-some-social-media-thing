# 13 · Build order, testing strategy, definition of done

## 1. Build order (each phase ends green: lint + typecheck + tests + its acceptance criteria)

### Phase 0 — Skeleton (docs 01, 02, 03 §1-3)
`bun` workspaces, biome, tsconfig graph, docker-compose, `@ve/config`, `@ve/core` (enums/ids/queues/formats/state machines), `@ve/db` schema + migrations + seed, pg-boss boot, pino. **Gate:** `bun run dev` starts all four processes against local PG/MinIO with zero errors; seed idempotent (run twice); CI pipeline green.

### Phase 1 — Radar (docs 03 §6, 04)
Connectors (reddit, youtube + fixtures for all), scouts, scoring (stat layer + rubric via mocked LLM in tests), clustering, digest, `@ve/telegram` sendDigest/sendAlert. **Gate:** doc 04 §7 criteria + dashboard-less validation via `GET /trends` (build the two API routes early).

### Phase 2 — Factory (docs 03 §8, 05)
`@ve/storage`, `@ve/media` (ffmpeg wrappers + fonts), `@ve/llm` full (structured runner, tts, transcribe, analyzeVideo, embed, metering), scriptwriter + similarity guard, compliance gate, asset jobs, render pipeline, clip pipeline (transcribe/analyze/cut), briefs lineage API route. **Gate:** doc 05 §7 criteria; cost ledger rows present for every step.

### Phase 3 — Approvals + Distribution (docs 06, 09, 11)
Approvals table flow, TG bot full (against a mock API in unit tests, live bot smoke manual), scheduler, Ayrshare connector (+ mock server for tests), publish execute/verify, engagement scan/reply, remaining API routes. **Gate:** doc 06 §8 + doc 09 §6 criteria.

### Phase 4 — Dashboard + Learning (docs 07, 10)
SPA all pages, auth, metrics snapshots, attribution, playbooks, costs rollup + budget guard, policy watch, ops endpoints. **Gate:** doc 07 §6 + doc 10 §5 criteria.

### Phase 5 — Production hardening (doc 12)
Provisioning, systemd, Caddy, backups, uptime monitoring, deploy script, real-credential smoke tests (one real post to each platform on a warm-up schedule), runbooks (`docs/runbooks/`: publish-failure, provider-blocked, restore-backup, rotate-keys). **Gate:** one fully-automated post published end-to-end from a real trend with TG approval, metrics arriving 24 h later.

Dependency-ordered PR slices are listed inside each phase's doc sections; keep PRs ≤ ~800 lines.

## 2. Testing strategy

| Layer | Tool | What |
|---|---|---|
| Unit | `bun test` | state machines (every allowed/blocked transition), similarity guard (crafted plagiarism/original pairs), scheduler slotting (caps/gaps/jitter bounds, fast-path), rights classifier post-processing (red-wins rollup), caption builders, cost calculators, connectors against fixtures |
| Integration | `bun test` + CI Postgres | migrations up from zero + seed; queue flows with real pg-boss (scout→score→cluster on fixtures; brief→script→assets→render with **tiny 2 s media fixtures** + real ffmpeg; approval decide race — two concurrent decides, one wins); API routes with in-process Hono `app.request()` |
| Contract mocks | Hono stub servers in-test | Ayrshare mock (create/history/analytics/comments), Telegram Bot API mock (sendVideo/editMessage capture), provider 429/5xx behavior |
| E2E happy path | `scripts/e2e-local.ts` | seeds fake trend → runs workers inline until posts `published` against mocks → asserts DB end-state + R2 objects exist (MinIO) |
| Manual smoke (phase 5) | checklist | real creds, one post per platform, TG approval on a phone, dashboard verify |

Test conventions: fixtures in `packages/*/fixtures/`; LLM calls in tests always mocked at the `@ve/llm` boundary (deterministic canned outputs per agent) except one live-optional suite behind `RUN_LIVE_LLM=1`; media fixtures generated by a script (`ffmpeg -f lavfi`) not committed binaries >100 KB.

## 3. Definition of done (system-level, maps to the research's Phase-1 exit)

1. Radar digest lands in TG twice daily for `ai-tech` with real Reddit+YouTube data.
2. A trend becomes a rendered ≥61 s captioned vertical video for ≤ $1.00 logged cost, blocked automatically if it plagiarizes or touches third-party rights.
3. TG approval (or dashboard) → scheduled → published via Ayrshare → permalink stored → snapshots at +3 h/+24 h/daily → visible charts on the post detail page.
4. Kill-switch, budget guard, policy watch, and dead-letter alerts all demonstrably fire.
5. Music category cannot publish; politics cannot auto-approve; no code path uploads third-party media.

## 4. Deferred (documented so nobody "helpfully" builds them early)

pgvector, SSE, Remotion templates, direct TikTok/YouTube APIs (audits), Whop API automation (submission stays manual), multi-admin/roles, X direct posting, music beds licensing, mobile app. Each has a note at its mention site in docs 02–11.
