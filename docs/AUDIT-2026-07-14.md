## Bottom line

Well-factored, unusually spec-literate code. **The safety-critical gates all hold**: nothing publishes third-party footage/music, the rights/similarity/music/radar-only checks fail closed, state changes go through row-locked transition helpers, the transactional approval decision is correctly serialized, and every LLM/paid call is metered.

But the tests invoke handlers **directly** and the fixtures were built around the same assumptions as the code, so a class of defect at the **seams** — pg-boss redelivery, the kill-switch, the edit-request cycle, the render pipeline, deploying without credentials — passes green while broken. Tallies: **12 High · 16 Medium · ~30 Low**. None architectural; each a small, localized fix. Failures fail toward stranded work, stale/single-frame media, silent dead-lettering, and one real auth hole.

## High-severity findings

### H1 · Login rate-limit is bypassable via `X-Forwarded-For` rotation — *security*
`apps/api/src/auth.ts:76`. The 5/min limiter keys on the client-supplied leftmost `x-forwarded-for` token. Rotate it and you are never limited — removing the only brute-force guard on the single admin password (8-char min, no lockout). **Reproduced live:** 10 rotating-XFF logins → 0 blocked; 10 fixed-XFF → 429 after 5.
**Fix:** key on the trusted right-most proxy hop / socket address, not the client-supplied left token.

### H2 · A crash mid-publish re-posts the same video — *correctness*
`apps/workers/src/engines/distribution/publish.ts:216-223`. A redelivered `publish.execute` that finds the post already `publishing` skips the whole guard-and-lock block (it only runs for `scheduled`) and calls `createPost` again — so a worker crash or the 900 s job-expiry between a successful send and the DB flip to `published` double-posts on retry. The documented `singletonKey: postId` backstop is a **no-op**: the repo creates pg-boss v10 *standard* queues, whose dedupe index `job_i1` only fires for policy `short` (verified in installed pg-boss 10.4.2 + a live double-send). The row-lock only covers the concurrent case, not crash-resume.
**Fix:** don't treat `publishing` as resume-and-re-send without re-locking; give publish queues a real dedupe policy.

### H3 · The kill-switch destroys in-flight work — *correctness*
`apps/workers/src/harness.ts:70-73`. The kill-switch throw sits *outside* the try/catch, so a gated job burns 3 retries (~8 min) then permanently dead-letters with **no alert**. A kill-switch is on for hours by nature, so every queued publish/approval/factory/engage job silently DLQs; scheduled posts strand; nothing re-arms on `/resume`. The documented revert-to-approved is unreachable through the real queue path (the gate throws first), and the kill-switch test calls the handler directly so it never sees this.
**Fix:** defer gated jobs (`RetryLater(15 min)`) instead of throwing into the retry budget.

### H4 · The edit-request cycle is broken in two independent places — *correctness*
- **a · media never changes:** asset/render idempotency is keyed to the *brief*, not the script version (`assets.ts:88`, `render.ts:334`), so v2 reuses v1 audio/captions/video. The requested change never reaches the render, yet the pre_publish caption-lint validates the clean v2 *text*.
- **b · no new card:** `approval.request` refuses to create the new row — its guard skips unless the prior approval is `rejected`/`expired`, but after an edit it is `edit_requested` (terminal). Posts sit in `draft` forever (`request.ts:61`).
**Fix:** key asset/render idempotency on `(brief, scriptVersion)`; treat `edit_requested` as recyclable in the request guard.

### H5 · Ken Burns slideshows render only scene 1, frozen — *correctness*
`packages/media/src/render.ts:55,69`. Each image is a looped stream (`-loop 1 -t per`) and `zoompan d=framesPer` emits `framesPer` frames *per input frame*, so image 1's chain alone exceeds the `-t audioDur` cap and the concat never reaches scenes 2..n. **Reproduced in the demo mp4:** frames at t=5s and t=65s are pixel-identical. Duration/dimension asserts pass, so the test and demo are blind.
**Fix:** feed one frame per image into `zoompan` (drop `-loop 1 -t per` on the kenBurns path).

### H6 · The AI-disclosure gate blocks every video after full spend — *correctness*
`apps/workers/src/engines/factory/compliance.ts:99` + `assets.ts:115`. Every TTS asset is labelled `licenseRef: "ai-gen:tts"`, and `checkAiDisclosure` fails pre_publish whenever an `ai-gen:*` asset exists and `script.aiDisclosure !== true`. Since every brief has TTS, any brief whose scriptwriter answered `false` (permitted by its own prompt) is blocked at the last gate after full spend — and that failure **revokes the earned auto-approval** for the whole (category, format).
**Fix:** gate on AI *visuals*, not the TTS narration.

### H7 · Uncredentialed production fabricates "published" — *deploy safety*
`packages/connectors/src/ayrshare.ts:100`. Fixture mode is `APP_ENV==='test' || !integrations[flag]`, so in *production* with `AYRSHARE_API_KEY` unset, `createPost` returns the success fixture, the post transitions to `published` with a fake permalink, and `publish.verify` "confirms" it. Workers never read `integrations`. Fine for reads; for writes it silently no-ops the product's output while reporting success.
**Fix:** refuse write-fixtures when `APP_ENV==='production'` (or guard `publish.execute` on `integrations.ayrshare`).

### H8 · Platform-rejection handling always throws — *correctness*
`apps/workers/src/engines/distribution/publish.ts:413`. `publish.verify` does `transitionPost(post, "failed")`, but `POST_TRANSITIONS.published = ["deleted"]` — every rejection throws `InvalidTransitionError` before the verbatim platform-error alert can fire, so the job dead-letters with a generic error instead of surfacing the policy signal.
**Fix:** allow `published → failed`, or transition before marking published.

### H9 · Licensed campaign clipping is unreachable end to end — *revenue path*
The day-1 revenue path can't be run through any production surface: the promote route resolves category only via `candidate.longFormId`, so campaign candidates always 422 (`apps/api/src/routes/longforms.ts:83`); nothing enqueues `clip.transcribe{kind:"campaign"}`; there is no upload route for `campaigns/{id}/source/`; the correct worker `promoteClipCandidate` is called only by tests.
**Fix:** resolve category from `campaigns.categoryId`; add the campaign source-upload + ingest path.

### H10 · Temp downloads are never deleted — *ops*
`packages/media/src/tmp.ts:20`. `downloadToTmp` writes into a shared `tmp/dl/` outside the per-job `tmpDir()` that gets cleaned; its "caller cleans" contract is honored by **zero of 10 callers**. The 24/7 worker leaks TTS mp3s, captions, scene images, and GB-scale clip sources until the disk fills.
**Fix:** download into each job's `tmpDir()`, or clean `tmp/dl/` in the same `finally`.

### H11 · First login bounces back to the login screen — *UX*
`apps/dashboard/src/router.tsx:70-86,177`. `Login.submit` sets the cookie and navigates but never clears the cached `["auth","me"]` 401; the Shell re-mounts, reads the cached error, and navigates back to `/login` before the refetch resolves. The second attempt works. Deterministic after every session expiry.
**Fix:** invalidate/reset the `["auth","me"]` query on successful login.

### H12 · Approval cards don't reliably reach Telegram — *approvals*
The interactive card with buttons is **never pushed on creation** — the worker only sends a plain text alert; `sendApprovalCard` is called solely by the bot's `/pending` (`request.ts:118`), so `tgMessageId` stays null and the reminder no-ops → approvals are effectively dashboard-only. Also the size gate assumes 50 MB but `sendVideo`-by-URL caps at 20 MB server-side, so a 20–50 MB render 400s and the card is lost (`cards.ts:26`).
**Fix:** push `sendApprovalCard` from the request worker and store `tgMessageId`; drop the URL-send threshold to 20 MB (thumbnail + link above it).

## How the system works today

A single-admin, 24/7 content operation in a Bun monorepo (Hono + Drizzle + Postgres + pg-boss + R2). Four engines on one queue substrate:

1. **Radar** — cron scouts pull Reddit/YouTube/X/TikTok → `raw_items` + snapshots → two-layer scoring (velocity z-scores vs 14-day baselines, then a Gemini rights-rubric) → embedding clustering into `trends` → an hourly Claude Editor-in-chief writes `briefs` under caps.
2. **Factory** — scriptwriter (idea only, never source text) + code similarity guard → blocking compliance gate → asset fan-out (TTS, Pexels stock, AI images, karaoke captions) → per-platform FFmpeg renders (61 s TikTok floor) → second compliance gate → approval request. A clip pipeline handles owned long-form and licensed campaigns.
3. **Distribution** — the approvals bridge auto-approves earned formats or parks the rest; the scheduler fills IST windows with cadence caps, ≥3 h gaps and jitter; `publish.execute` sends per-platform Ayrshare payloads with AI flags; `publish.verify` confirms the permalink; an engagement agent scans and gate-replies to comments.
4. **Learning** — snapshots at +3 h/+24 h then tapering; a weekly attribution job builds deterministic tables (medians, Spearman, deciles) → a Claude analyst → a per-category playbook draft (human-approved) + a kill-list that revokes auto-approve for losing formats.

The substrate is a pg-boss harness (payload validation, kill-switch gate on the publish/factory/engage set, final-retry alerts, a budget guard, a monthly policy-watch). Surfaces: the Hono API (`/api/v1`, session + bearer), a React dashboard (9 pages), and a grammY Telegram bot. Every write goes through the API; bytes live in R2/MinIO behind presigned URLs; every LLM/paid call is metered.

## How to test it locally

Tuned to this machine: no Docker CLI, but native Postgres 18 (`:5432`, role `ve/ve`, DBs `viral_engine` + `viral_engine_test`), native MinIO (`:9000`), and FFmpeg 8.1 with libass are present. Port 3000 is taken by another project — pass `API_PORT`.

```bash
# 1. Static gates (Postgres up; tests hit viral_engine_test via .env.test)
bun install
bunx tsc -b
bunx biome check .
bun test
bun run db:check
bun run build:dashboard

# 2. Offline end-to-end, no credentials — writes a real mp4 to MinIO
bun run scripts/radar-demo.ts
bun run scripts/factory-demo.ts
bun run scripts/distribution-demo.ts

# 3. Run the stack + smoke it
bun run db:migrate && bun run db:seed
API_PORT=3210 bun run dev      # api :3210 · dashboard :5173 · workers · bot
TOKEN="change-me-to-a-long-random-token"
curl -s localhost:3210/healthz
curl -s -H "authorization: Bearer $TOKEN" "localhost:3210/api/v1/trends?category=ai-tech"
```

Verified live this session: `/healthz` reports db/boss/storage OK and flags stale workers; unauthenticated routes 401; the presign endpoint rejects `../../etc/passwd` with 403; a music-trend brief returns 422; login issues a session; the demo render round-trips through MinIO via presigned GET (HTTP 206). The bot idles without a Telegram token.
